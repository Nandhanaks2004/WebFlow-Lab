Lab 12

HTTP vs HTTPS

🔓
HTTP

Data travels in plain text. Anyone intercepting traffic can read it. No certificate required.

INSECURE
🔒
HTTPS

Data is TLS/SSL encrypted. Interceptors see gibberish. Requires SSL certificate from CA.

SECURE
📦 Packet Interception Demo
HTTP — Plain text (attacker can read):
Intercepted → POST /login · user=alice&pass=password123
HTTPS — Encrypted (attacker sees noise):
Intercepted → TLSv1.3 · 🔒 EncryptedData
TLS Handshake: Client and server negotiate cipher suite → exchange certificates → generate shared secret → all future data encrypted. Even if intercepted, attacker cannot decrypt without the session key.

Lab 14

Security Thinking

🛡️ Trust Boundary Map
🌍 Public Zone
User inputs, URLs, cookies, headers — ALL untrusted. Anything from the client can be manipulated.
🚧
🔒 Trusted Zone
Server-side code, database, session store — validate everything before it reaches here.
🤔 Security Questions — Click to Reveal Answers
INJECTION
Where does user input enter the application? Where is it dangerous?
User input enters through: URL parameters, form fields, HTTP headers, cookies. It becomes dangerous when it reaches: SQL queries (SQL injection), HTML output (XSS), OS commands (command injection), file paths (path traversal). RULE: Never trust user input. Always validate, sanitize, and escape.
AUTH
Can a user change their session cookie to access another account?
YES — if session IDs are predictable or not properly validated. Session IDs must be: cryptographically random (e.g. 128-bit), stored server-side mapped to user, validated on every request. If an attacker can guess or steal a session ID (session hijacking), they impersonate that user.
HTTP
What happens when a GET request carries sensitive data in the URL?
GET parameters appear in: browser history, server access logs, HTTP Referer headers, proxy logs. NEVER put passwords, tokens, or PII in GET parameters. Use POST with a request body + HTTPS for sensitive data transmission.
HTTPS
Does HTTPS protect against all attacks?
NO. HTTPS only encrypts data in transit. It does NOT protect against: server-side bugs, XSS (attacker runs code in-browser), CSRF (forged requests), stolen credentials, or compromised server. HTTPS is necessary but not sufficient for security.
COOKIES
What makes cookies secure or insecure?
Secure cookies use: HttpOnly flag (JS can't read = XSS safe), Secure flag (HTTPS only), SameSite=Strict (CSRF protection), short expiry. Insecure cookies: accessible to JS, sent over HTTP, no SameSite, stored with predictable values.
